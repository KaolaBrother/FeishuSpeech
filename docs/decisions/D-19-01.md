# D-19-01: Sleep/wake recovery contract

- Status: Accepted
- Date: 2026-06-13
- Issues: #19
- Affected: FeishuSpeech/App/AppDelegate.swift, FeishuSpeech/ViewModels/MainViewModel.swift, FeishuSpeech/Services/HotKeyService.swift, FeishuSpeech/Services/FeishuAPIService.swift, FeishuSpeechTests/MainViewModelTests.swift, FeishuSpeechTests/HotKeyServiceTests.swift, FeishuSpeechTests/FeishuAPIServiceTests.swift

## Problem

macOS sleep can suspend the app while recording, transcribing, or holding stale
CGEventTap state. After wake, the app must not finish a pre-sleep transcription,
keep showing stale overlay UI, reuse a stale Feishu token or network error, or
believe an Event Tap is healthy when the system disabled or removed it.

The lifecycle notification may also arrive before `AppDelegate` has received the
`MainViewModel` instance, so sleep/wake events need a deterministic delivery
contract during startup.

## Decision

`AppDelegate` registers `NSWorkspace.willSleepNotification` and
`NSWorkspace.didWakeNotification` with `NSWorkspace.shared.notificationCenter`.
The observer tokens are stored and removed during application termination. If a
workspace lifecycle event arrives before `setViewModel(_:)`, `AppDelegate`
queues the event and replays queued events to the view model after injection.

`MainViewModel.handleSystemWillSleep()` and
`MainViewModel.handleSystemDidWake()` share the same coordinator reset path:
they cancel any in-flight transcription by advancing the transcription
generation and cancelling the task, hide the overlay, force-clean audio
resources, stop the max-duration timer, clear the consecutive-failure counter,
set status to `.idle`, and reset `HotKeyService` to idle. Both handlers call
`FeishuAPIService.resetStateForWake()`. The wake handler additionally calls
`HotKeyService.recoverAfterWake()`.

`HotKeyService.recoverAfterWake()` first cancels stale wake input by cancelling
pending transitions, returning the hotkey state to `.idle`, and clearing
`previousFlags`. It then checks whether a real Event Tap exists and whether
`CGEvent.tapIsEnabled(tap:)` reports it enabled. A present, enabled tap remains
active. A missing or disabled tap is recovered through the normal
`stopMonitoring()` / `startMonitoring()` lifecycle instead of mutating tap state
in place.

`FeishuAPIService.resetStateForWake()` clears the cached token, token expiry,
and last network error, then marks network availability true so post-wake work
can perform fresh network checks.

## Alternatives considered

- Rely on normal Event Tap disable callbacks after wake -- rejected; wake may
  leave the app with stale tap state before another callback is delivered.
- Keep pre-sleep transcription alive -- rejected; recorded audio and network
  state can be stale after sleep and should not be replayed into the active
  cursor.
- Only clear the API token on explicit authentication failures -- rejected;
  sleep/wake can invalidate network assumptions independently of HTTP status.

## Consequences

- Sleep and wake both return the coordinator to an idle, non-transcribing state.
- Wake recovery refreshes Feishu API token/network state before later requests.
- Missing or disabled Event Taps are restarted through the same lifecycle used
  by normal monitoring start/stop.
- DEBUG test hooks expose only non-secret recovery evidence: hotkey wake
  recovery returns a restart count, and API wake snapshots expose token/expiry
  presence, network-error description, and network availability rather than
  credential or token values.
