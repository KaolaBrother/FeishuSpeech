# D-13-01: TextInputSimulator clipboard-restore, overlayMessage model state, run-loop mode, and overlay generation guard

- Status: Accepted
- Date: 2026-06-12
- Issues: #13, #14, #16, #17
- Affected: FeishuSpeech/Services/TextInputSimulator.swift, FeishuSpeech/ViewModels/MainViewModel.swift, FeishuSpeech/Services/HotKeyService.swift, FeishuSpeech/UI/OverlayWindowController.swift

## Problem

Four distinct races and silent-failure bugs surfaced together in this bundle:

- #13 (P1-high): `TextInputSimulator` wrote the transcribed text to `NSPasteboard.general`,
  sent a synthetic Cmd+V, then immediately restored the previous clipboard contents. Because
  `CGEvent` keyboard delivery is asynchronous, the restore could execute before the target
  application had read the paste, clobbering the clipboard with the old data before the paste
  completed. Additionally, only the first pasteboard item's `string` was saved and restored,
  silently dropping non-string types (images, RTF, etc.).
- #14 (P1-high): When the Feishu speech API returned a non-error response with an empty or
  whitespace-only transcript, `MainViewModel` performed no action and emitted no user-visible
  signal. The user saw the recording indicator disappear with no explanation, making it
  indistinguishable from a successful recognition with empty speech.
- #16 (P1-high): `HotKeyService` schedules `maxDurationTimer` on the default run-loop mode
  (`RunLoop.main`). When the macOS menu-bar menu is open, the run loop switches to
  `NSEventTrackingRunLoopMode`, which does not execute timers scheduled in `.default`. As a
  result, the recording time limit was silently skipped whenever the user had the menu open
  while recording, allowing unbounded recording sessions.
- #17 (P1-high): `OverlayWindowController` called `NSAnimationContext.runAnimationGroup` to
  fade in a new overlay message. If a hide-completion block from a prior `hide()` call fired
  after the new `show()` had already started, it closed the window mid-animation, making the
  overlay flash and disappear immediately.

## Decision

**(a) changeCount-based paste confirmation and full pasteboard snapshot (#13).**
`TextInputSimulator` captures a full snapshot of the pasteboard before writing: it reads all
`types` from `NSPasteboard.general` and stores a `[(type: NSPasteboard.PasteboardType,
data: Data)]` array. After sending the synthetic Cmd+V, the simulator polls
`NSPasteboard.general.changeCount` at a short interval (up to a bounded timeout) to detect
that the target application has read from the pasteboard (changeCount increments on each
read). Only after the changeCount advances — or the timeout expires — is the full snapshot
written back to restore all types and data. If the changeCount does not advance within the
timeout, a fallback path posts an `NSNotification` so the caller can surface a warning.

**(b) `overlayMessage` observable model state for empty results (#14).**
`MainViewModel` adds `@Published var overlayMessage: String?`. When the API returns an empty
transcript, instead of silently returning, `MainViewModel` sets `overlayMessage` to a
localised string (e.g. "未识别到语音内容"). `OverlayWindowController` is the intended
consumer of this property. The overlay display path is kept separate from the
`TextInputSimulator` path so empty-result feedback does not interfere with a real paste.

Note: as of this ADR, `overlayMessage` is published on `MainViewModel` but is not yet wired
to a visible UI element. See R1 in Consequences.

**(c) `.common` run-loop mode for `maxDurationTimer` (#16).**
`HotKeyService` schedules `maxDurationTimer` using `RunLoop.main.add(timer, forMode: .common)`
instead of the default `.default` mode. `.common` is a set that includes both
`.default` and `NSEventTrackingRunLoopMode`, ensuring the timer fires regardless of whether
a menu or other tracking event loop is active.

**(d) Generation counter in `OverlayWindowController` (#17).**
`OverlayWindowController` maintains a monotonically incrementing `Int` generation counter.
Each call to `show()` increments the counter and captures the new value in the animation
completion closure. Each call to `hide()` similarly captures the current generation. Any
completion block that fires with a stale (mismatched) generation is a no-op and does not
close or alter the window. This prevents a hide-completion from a superseded call from
closing a window that a newer `show()` has already claimed.

## Alternatives considered

- Delay the restore with a fixed `DispatchQueue.asyncAfter` instead of changeCount polling —
  rejected; a fixed delay is fragile under system load and wastes time on fast targets.
- Use `NSPasteboard.writeObjects` with a `NSPasteboardWriting` proxy that vends data lazily —
  rejected; lazy-promise pasteboard items require the app to remain alive during the read,
  which is unreliable in a menu-bar app context.
- Surface empty results by posting a system notification banner — rejected for #14; a banner
  is heavyweight and requires `NSUserNotificationCenter` authorization; the overlay path is
  already established for transient status messages.
- Use `DispatchSourceTimer` instead of `RunLoop` timer for max duration (#16) — viable
  alternative, but `.common` mode is the minimal, idiomatic fix and avoids introducing a
  second timer primitive.
- Cancel all pending hide calls in `show()` via a `DispatchWorkItem` token — also viable for
  #17, but a generation counter is simpler and does not require retaining a work-item token
  across async hops.

## Consequences

- **#13**: Clipboard content (all types) is preserved across a synthetic paste. The
  changeCount confirmation adds at most one polling interval of latency before restore; this
  is imperceptible in normal use. The `[(type:data:)]` snapshot approach correctly handles
  multi-item and non-string pasteboard contents.
- **#14**: Empty API results are no longer silent. `overlayMessage` is an observable property
  on `MainViewModel` available to any future UI consumer.
- **#16**: `maxDurationTimer` fires reliably regardless of menu-bar tracking mode. The change
  is a one-line scheduling mode update; no other timer behaviour is altered.
- **#17**: Overlay show/hide sequences are race-free. The generation counter adds one `Int`
  field to `OverlayWindowController` and zero async overhead.

### Follow-up items

- **R1 (open)**: `overlayMessage` on `MainViewModel` is not yet wired to a visible UI
  element. A future issue should connect it to the overlay or status bar so empty-result
  feedback is actually shown to the user.
