# D-11-01: Feishu API direct-response, token-expiry, and cancellation contract

- Status: Accepted
- Date: 2026-06-13
- Issues: #11, #12, #21
- Affected: FeishuSpeech/Services/FeishuAPIService.swift, FeishuSpeech/Models/SpeechResult.swift, FeishuSpeechTests/FeishuAPIServiceTests.swift, FeishuSpeech.xcodeproj/project.pbxproj

## Problem

Three API-path defects shared the same retry/direct-request surface:

- #11: `DirectFeishuHTTPClient` waited for connection close before parsing a
  response. Feishu responses with complete `Content-Length` or terminal chunked
  bodies could sit buffered until timeout when the peer kept the connection open.
- #12: tenant token caching ignored Feishu's `expire` value and always used a
  hardcoded 6000 second lifetime.
- #21: `CancellationError` could pass through generic retry/direct-IP handling,
  causing cancelled work to continue into retries, later IP attempts, or DNS
  fallback.

## Decision

**Direct HTTP responses are completeness-aware.** The direct client now attempts
to parse buffered data after each receive. `Content-Length` responses complete
when the declared byte count is present, and chunked responses complete when the
terminal zero-size chunk plus trailer terminator is present. Close-delimited
responses still wait for connection close. On timeout, the buffered parse is
attempted once before throwing `APIError.timeout`.

**Token lifetime comes from Feishu `expire` with a 300 second safety margin.**
`AuthResponse` decodes optional `expire` values from either JSON numbers or
numeric strings. Positive values use `expire - 300`; missing, zero, or negative
values retain the legacy 6000 second fallback. Short positive values intentionally
do not fall back to 6000 seconds, so an `expire` value below the safety margin
forces immediate refresh behavior.

**Cancellation is terminal.** `withRetry` and `sendDirectRequest` check task
cancellation at loop boundaries and rethrow `CancellationError` before retriable
or generic error handling. A cancelled direct-IP request does not advance to the
next IP and does not enter URLSession fallback.

**API tests are project-wired, with one explicit existing exclusion.**
`FeishuSpeechTests` is now a runnable macOS unit-test target for the focused API
regression tests and the existing passing suites. `AudioRecorderRecoveryTests.swift`
remains excluded because it is a reproduced pre-existing AudioRecorder blocker
outside this API bundle.

## Alternatives considered

- Continue relying on `Connection: close` to delimit direct responses -- rejected;
  it preserves the timeout bug whenever a complete body is buffered before close.
- Parse buffered timeout data with close-delimited semantics -- rejected; timeout
  fallback must only accept responses that prove body completeness.
- Clamp short positive `expire` values to the 6000 second fallback -- rejected;
  Feishu supplied a positive expiry, and extending it past the reported lifetime
  risks using stale credentials.
- Treat `CancellationError` as a retriable generic error -- rejected; caller
  cancellation must be observable and must stop network fan-out.
- Re-enable every existing test file while wiring the target -- rejected for this
  bundle because `AudioRecorderRecoveryTests.swift` fails for an unrelated,
  pre-existing AudioRecorder issue.

## Consequences

- Direct Feishu HTTP requests no longer depend on peer connection close when a
  framed body is already complete.
- Timeout handling can still succeed for fully buffered framed responses, while
  incomplete bodies remain timeouts.
- Tenant token refresh timing follows Feishu's reported expiry with a fixed
  300 second safety margin.
- Cancellation now bounds the retry/direct-IP loop to the caller's cancellation
  decision.
- The API recovery behavior is covered by `FeishuAPIServiceTests`; the remaining
  excluded AudioRecorder test file should be handled by a future AudioRecorder
  issue before it is re-enabled.
