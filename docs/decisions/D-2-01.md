# D-2-01: Network resilience and cancellation for FeishuAPIService

- Status: Accepted
- Date: 2026-06-11
- Issues: #2, #3, #4
- Affected: FeishuSpeech/Services/FeishuAPIService.swift

## Problem

The direct-IP HTTP client in FeishuAPIService had three resilience gaps:
- #2 (P1): NWConnection .waiting state ignored — unreachable CDN IPs hang ~30–75s each.
- #3 (P1): 5 hardcoded Feishu CDN IPs with no DNS fallback — app breaks permanently if IPs rotate.
- #4 (P0): the 30s transcription timeout cancelled the Swift task but never cancelled the
  in-flight NWConnection — "识别中" stuck up to ~150s (5 IPs × 30s).

## Decisions

### #2 — abort on .waiting
Add `case .waiting(let error): finish(.failure(error))` to stateUpdateHandler in
DirectFeishuHTTPClient.send(). Routes through the existing finish() guard (NSLock + isFinished,
which also calls connection.cancel()) so the continuation resumes immediately. Propagates the
real NWError for diagnostic fidelity.

### #4 — propagate cancellation to NWConnection
Wrap send()'s withCheckedThrowingContinuation in withTaskCancellationHandler. Introduce
CancelBox (@unchecked Sendable, NSLock-protected) holding the connection + an abort closure.
onCancel calls cancelBox.cancel() → connection.forceCancel() + finish(.failure(CancellationError())).
CancelBox handles the already-cancelled-at-entry race by checking cancelled flag on store().
All resumes go through the single finish() guard — no second resume path.

### #3 — URLSession DNS fallback
After all 5 direct IPs fail, call sendViaURLSession(path:headers:body:) against
https://open.feishu.cn using URLSession.shared. No ATS exceptions or entitlement changes
required (macOS 13+, HTTPS). URLSession.shared.data(for:) handles Swift task cancellation
natively. Maps HTTPURLResponse → existing DirectHTTPResponse(statusCode:body:).

### MainViewModel
No change required. Existing withTimeout(30s) + group.cancelAll() already issues cancellation;
#4 makes the service honor it.

## Alternatives considered
- #2: synthesize APIError.timeout — rejected (loses real NWError diagnostics).
- #4: resume continuation directly in onCancel — rejected (double-resume hazard).
- #4: poll Task.isCancelled inside receiveLoop — rejected (must abort suspended wait, not next callback).
- #3: custom URLSession with pinned config — rejected (unnecessary complexity, direct-IP remains primary path).
- #3: live DNS refresh before each IP attempt — rejected (URLSession fallback already handles this).

## Consequences
- Unreachable IPs abort in milliseconds (#2); permanent IP-rotation breakage eliminated (#3);
  worst-case "识别中" bounded by 30s instead of ~150s (#4).
- CancelBox (@unchecked Sendable, NSLock) is the only new type; send() restructured but logic unchanged
  except .waiting and cancel registration.
- forceCancel() skips TLS close_notify — acceptable for aborting unestablished connections.
