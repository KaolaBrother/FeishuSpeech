# D-18-01: Feishu credential Keychain storage contract

- Status: Accepted
- Date: 2026-06-13
- Issues: #18
- Affected: FeishuSpeech/Models/AppSettings.swift, FeishuSpeech/Services/KeychainCredentialStore.swift, FeishuSpeech/Views/SettingsView.swift, FeishuSpeech/ViewModels/MainViewModel.swift, FeishuSpeechTests/AppSettingsCredentialStorageTests.swift

## Problem

`AppSettings` previously allowed Feishu API credentials to live in user defaults,
either inside the encoded `FeishuSpeechSettings` payload or as standalone
`appId` / `appSecret` default keys. That made the App Secret part of a normal
preferences store instead of a credential store.

The fix also needed to preserve existing installations: users who already had
credentials in user defaults should not lose them if Keychain migration, Keychain
read, or Keychain write fails.

## Decision

Feishu API credentials are now stored through `CredentialStoring`, whose default
implementation is `KeychainCredentialStore`.

`KeychainCredentialStore` stores macOS Security.framework generic password items
using service `Siji.FeishuSpeech.credentials` and account values matching
`CredentialAccount.appId.rawValue` and `CredentialAccount.appSecret.rawValue`.
Read, save, and delete operations use `SecItemCopyMatching`, `SecItemUpdate`,
`SecItemAdd`, and `SecItemDelete`.

`AppSettings` keeps `appId` and `appSecret` as runtime properties, but its custom
`Codable` implementation no longer encodes those fields. The encoded
`FeishuSpeechSettings` payload contains only non-credential preferences:
`autoInsert`, `playSound`, and `launchAtLogin`.

On load, `AppSettings` attempts to migrate legacy credentials from both supported
legacy locations:

- encoded `FeishuSpeechSettings` fields `appId` and `appSecret`;
- standalone user-default keys `appId` and `appSecret`.

Standalone values win when both legacy sources are present. Legacy defaults are
scrubbed only after migration succeeds and both credentials can be read from the
credential store. If migration or credential read fails, `AppSettings` returns
the legacy values as a fallback and marks that fallback so a later `save()` does
not rewrite or delete the only remaining credential copy.

`SettingsView` no longer binds Feishu credentials through `@AppStorage`. It uses
transient `@State` values loaded from `MainViewModel.settings` and persists them
through `MainViewModel.updateSettings(...)`, which delegates to `AppSettings.save()`.

## Alternatives considered

- Keep App Secret in `UserDefaults` and document the risk -- rejected; the field
  is an API credential and should use the platform credential store.
- Migrate once and always scrub legacy defaults -- rejected; a Keychain write or
  read failure could erase the user's only recoverable credential copy.
- Bind settings UI directly to user defaults -- rejected for credentials; that
  would bypass the Keychain-backed save path.

## Consequences

- New saves write Feishu App ID and App Secret to the credential store instead of
  encoding them in `FeishuSpeechSettings`.
- `FeishuSpeechSettings` remains the preferences payload for non-secret settings.
- Existing user-default credentials are migrated opportunistically and removed
  only after a successful credential-store round trip.
- The migration and failure-preservation behavior is covered by
  `AppSettingsCredentialStorageTests`.
