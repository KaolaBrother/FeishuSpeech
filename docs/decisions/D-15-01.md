# D-15-01: Recorder fail-fast failures and runtime microphone refresh

- Status: Accepted
- Date: 2026-06-13
- Issues: #15
- Affected: FeishuSpeech/Services/AudioRecorder.swift, FeishuSpeech/ViewModels/MainViewModel.swift, FeishuSpeech/Services/PermissionManager.swift, FeishuSpeech/App/AppDelegate.swift, FeishuSpeechTests/MainViewModelTests.swift, FeishuSpeechTests/PermissionManagerTests.swift

## Problem

Mid-recording `AVCaptureSession` failures and repeated audio conversion failures could leave
the recorder looking active until the normal stop/transcribe path ran. That delayed failure
reporting and risked surfacing a misleading downstream error such as empty audio instead of the
actual capture failure.

The app also refreshed accessibility and secure-input state at runtime, but microphone
authorization could remain stale after the user changed it in System Settings. Rechecking
microphone authorization must not show a permission prompt during the periodic poll.

## Decision

`AudioRecorder` publishes a specific `RecordingFailure` when capture cannot continue:

- `.runtime` for `AVCaptureSession.runtimeErrorNotification`
- `.interrupted` for `AVCaptureSession.wasInterruptedNotification`
- `.deviceLost` for `AVCaptureDevice.wasDisconnectedNotification`
- `.formatConversion` after the conversion-error counter reaches `maxConversionErrors`

The abort path always hops to the main queue before calling `forceCleanup()` and mutating
`@Published` state when the notification or sample-buffer path arrives off-main. `forceCleanup()`
continues to hand `AVCaptureSession.stopRunning()` and session input/output removal to
`sessionQueue`; the main hop is only for published state mutation and cleanup coordination.

`MainViewModel` subscribes to `audioRecorder.$failure`. When a failure arrives, it hides the
overlay, force-cleans the recorder, stops the max-duration timer, sets `status` to the failure's
localized message, and puts `HotKeyService` into `.error` with the same message. The failure path
does not call `stopRecording()` or start transcription.

`PermissionManager` adds `refreshMicrophoneStatus()`, which maps the current
`AVCaptureDevice.authorizationStatus(for: .audio)` to `microphoneGranted` without calling
`requestAccess`. The authorization-status lookup is injectable in Debug builds for tests.
`AppDelegate` calls this refresh in the existing 2-second permission poll alongside
`refreshAccessibilityStatus()` and `refreshSecureInputStatus()`.

## Alternatives considered

- Let the normal Fn-release or max-duration path discover the problem - rejected because it
  reports too late and can produce a misleading empty-audio or permission-style error.
- Stop the capture session synchronously on the main actor during abort - rejected because
  `AVCaptureSession.stopRunning()` can block; session stop remains owned by `sessionQueue`.
- Reuse `checkAllPermissions()` for periodic microphone refresh - rejected because it can prompt
  for microphone access when status is `.notDetermined`; runtime polling must be no-prompt.

## Consequences

- Capture failures are user-visible immediately with the concrete recording failure reason.
- Recorder cleanup and `@Published` mutations remain main-thread safe even when AVFoundation
  notifications arrive on a background thread.
- Session stop semantics from issue #9 are preserved: blocking AVFoundation session work stays
  off the main actor.
- The app notices microphone permission changes during the same 2-second poll that already
  refreshes accessibility and secure-input status.
- Tests cover conversion exhaustion, runtime failure publication, background failure cleanup on
  main, `MainViewModel` failure handling, microphone refresh, and test-state reset hooks.
