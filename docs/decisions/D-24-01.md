# D-24-01: Hot-key monitoring auto-clear policy

- Status: Accepted
- Date: 2026-06-13
- Issues: #22, #23, #24
- Affected: FeishuSpeech/Services/HotKeyService.swift, FeishuSpeech/ViewModels/MainViewModel.swift, FeishuSpeechTests/MainViewModelTests.swift

## Problem

`HotKeyService.monitoringState` now exposes tap health through `.stopped`, `.active`, and
`.failed(TapFailureReason)`. `MainViewModel` must surface tap failures to the user, recover
when the tap becomes active again, and avoid clearing unrelated errors that happen to be
visible at the same time.

The teardown path also needs to preserve the subscription invariant from the hot-key state
machine work: shutdown must release the dedicated `$state` subscriber and must not leave a
stale sink alive after `cleanup()`.

## Decision

`MainViewModel` treats every `.failed` monitoring state as the same user-facing hot-key
monitoring error:

`热键不可用，请检查辅助功能权限`

When `monitoringState` later becomes `.active`, `MainViewModel` auto-clears only that exact
stale hot-key monitoring error. The implementation records that the displayed error came
from monitoring failure, then clears to `.idle` only if the current status is still
`.error("热键不可用，请检查辅助功能权限")`. If another error is current, `.active` leaves it
unchanged.

For teardown, `MainViewModel.cleanup()` routes through `stopHotKeyMonitoring()`.
`stopHotKeyMonitoring()` nils `stateCancellable` before calling `HotKeyService.stopMonitoring()`,
so app cleanup and ordinary monitoring shutdown share the same subscriber-release invariant.

## Alternatives considered

- Clear every `.error` when monitoring becomes `.active` - rejected because speech
  recognition, recording, or configuration errors are unrelated to CGEventTap recovery.
- Leave the stale hot-key monitoring error visible until the general error debounce clears it
  - rejected because tap recovery has an explicit `.active` signal and should remove only the
  stale tap-failure message immediately.
- Let `cleanup()` call `HotKeyService.stopMonitoring()` directly - rejected because it would
  bypass the `stateCancellable` release contract owned by `stopHotKeyMonitoring()`.

## Consequences

- Accessibility-not-trusted and tap-creation failures both surface as
  `热键不可用，请检查辅助功能权限`.
- Successful tap recovery removes only that stale monitoring error.
- Unrelated errors remain visible when monitoring recovers.
- Future teardown changes must keep `cleanup()` routed through `stopHotKeyMonitoring()` unless
  they provide an equivalent explicit release of `stateCancellable`.
