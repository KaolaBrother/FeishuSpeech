# D-6-01: HotKeyService/MainViewModel state-machine contract

- Status: Accepted
- Date: 2026-06-11
- Issues: #6, #7, #8
- Affected: FeishuSpeech/Services/HotKeyService.swift, FeishuSpeech/ViewModels/MainViewModel.swift

## Problem

`HotKeyService` and `MainViewModel` maintained two loosely coupled state machines that drifted
out of sync under concurrent input, producing three distinct bugs:

- #6 (P1): `handleFnReleased` had no guard on current state — calling it from `.transcribing`,
  `.error`, `.cancelled`, or `.idle` would re-enter the transcription path, starting a new
  recording session over an active one.
- #7 (P1): The max-duration timer called `stopRecordingAndTranscribe()` directly, bypassing the
  state machine. If Fn was released after the timer fired, `stopRecordingAndTranscribe()` ran
  twice — the second call saw no audio data and emitted a spurious error.
- #8 (P1): `startHotKeyMonitoring` appended a new `$state` subscriber into the shared
  `cancellables` set on each call without cancelling the previous one. Multiple start/stop
  cycles accumulated live subscribers, causing duplicate event dispatch.

## Decision

Four contract rules enforced across `HotKeyService` and `MainViewModel`:

**(a) `handleFnReleased` is a no-op except in `.recording`.**
All other states (`.transcribing`, `.error`, `.cancelled`, `.idle`) are explicit no-ops with
no state transition. Only `.recording → .transcribing` is the normal Fn-release path.

**(b) Max-duration stop routes through `forceTranscribing()`.**
`HotKeyService` exposes `forceTranscribing()`, which transitions `.recording → .transcribing`
through the state machine. `handleMaxDurationReached` calls only `forceTranscribing()`.
`stopRecordingAndTranscribe()` is made idempotent so the subsequent Fn-release no-op in rule
(a) produces no error even if called after the timer has already stopped the session.

**(c) Stale transcription errors are discarded when a new session is active.**
In `transcribeAudio`'s catch block, errors are discarded when `hotKeyService.state.isActive`
is true (a new session has already started), preventing errors from a cancelled session from
surfacing against the new session.

**(d) `$state` subscription is owned by a dedicated `stateCancellable` property.**
`MainViewModel` holds `stateCancellable: AnyCancellable?` as a named stored property.
`startHotKeyMonitoring` assigns to it (implicitly cancelling any prior subscriber).
`stopHotKeyMonitoring` nils it before calling `stopMonitoring()`. At most one live `$state`
subscriber exists at any time.

## Alternatives considered

- Gate `handleFnReleased` with a single `guard state == .recording` — accepted as rule (a).
- Re-use the existing `cancellables: Set<AnyCancellable>` for the `$state` subscription —
  rejected; set insertion provides no way to cancel a specific prior subscriber without
  cancelling all subscriptions.
- Cancel all `cancellables` in `stopHotKeyMonitoring` — rejected; `cancellables` holds
  unrelated Combine chains; mass-cancellation would break other observers.
- Make `forceTranscribing()` a no-op outside `.recording` — accepted as part of rule (b);
  idempotency means duplicate calls are safe.

## Consequences

- Concurrent session bugs from Fn-release during active transcription are eliminated (#6).
- Max-duration auto-stop no longer produces a spurious double-stop error (#7).
- Multiple `startHotKeyMonitoring` calls no longer accumulate live subscribers (#8).
- `forceTranscribing()` is now a public surface on `HotKeyService`; callers outside the
  max-duration path must not use it (they should call `handleFnReleased` instead).
- `stopRecordingAndTranscribe()` idempotency must be preserved in future refactors.
