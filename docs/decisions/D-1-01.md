# D-1-01: AudioRecorder session recovery contract

- Status: Accepted
- Date: 2026-06-11
- Issues: #1
- Affected: FeishuSpeech/Services/AudioRecorder.swift

## Problem

`AudioRecorder.startRecording()` checked `isRecording` before performing any cleanup. If a
previous recording session ended via a cancel or error path without properly clearing the flag,
`isRecording` remained `true` and the guard returned early, permanently blocking any new
recording until the app was restarted. The stale flag also meant the underlying
`AVCaptureSession` could be left in an inconsistent running state.

## Decision

Establish a three-part recovery contract enforced by a single `forceCleanup()` helper:

1. **startRecording()** — call `forceCleanup()` before the `isRecording` guard so every
   recording attempt starts from a clean, known state regardless of how the previous session
   ended.
2. **handleCancelledState() / handleErrorState()** — call `forceCleanup()` in each exit path
   so `isRecording` and the `AVCaptureSession` are always torn down together when a session
   ends abnormally.
3. **resetService()** — call `forceCleanup()` so a deliberate reset from the UI or a test
   reliably recovers a stuck mic with a single call.

`forceCleanup()` must remain idempotent: calling it on an already-stopped session must be
a no-op with no side effects.

## Alternatives considered

- Guard removal only (skip the `isRecording` check entirely) — rejected; the guard still
  prevents concurrent double-starts; removing it would require a separate mutex.
- Reset `isRecording` flag only, without stopping the session — rejected; leaves the
  `AVCaptureSession` running and wastes mic resources.
- Restart the session on next call instead of stopping on cancel/error — rejected; does not
  address the flag inconsistency and adds hidden state to the caller contract.

## Consequences

- `isRecording` is always consistent with the actual `AVCaptureSession` running state.
- A stuck mic is recoverable at runtime via `resetService()` without an app restart.
- `forceCleanup()` must stay idempotent; any future changes to teardown logic must preserve
  that invariant.
- Cancel and error paths now both call the same teardown code, reducing duplication and the
  risk of divergence.
