# D-5-01: HotKeyService tap-lifecycle and threading contract

- Status: Accepted
- Date: 2026-06-11
- Issues: #5, #9, #10
- Affected: FeishuSpeech/Services/HotKeyService.swift, FeishuSpeech/Services/PermissionManager.swift, FeishuSpeech/ViewModels/MainViewModel.swift

## Problem

Three co-located defects in the `HotKeyService` CGEventTap machinery surfaced together and
required coordinated fixes to the threading model, the failure-visibility surface, and the
secure-input guard:

- #5 (P0-critical): A failed or invalidated CGEventTap produced no observable signal. Callers
  had no way to distinguish a healthy tap from a silently dead one. The retry policy also had a
  hard 3-attempt cap, after which the service stopped attempting recovery permanently.
- #9 (P1-high): The CGEventTap `CFRunLoopSource` was added to `CFRunLoopGetMain()`. Because
  `AVCaptureSession.startRunning()` also blocks the main run loop, the two competed for main
  thread time, causing sporadic hotkey events to be dropped or delayed. Additionally,
  `previousFlags` was not reset on `stopMonitoring()`, so a restart could inherit a stale
  Fn-down state and trigger a phantom recording.
- #10 (P2-medium): macOS `IsSecureEventInputEnabled()` silently suppresses CGEventTap keydown
  events when secure keyboard entry is active (e.g. Terminal, 1Password). The app gave no
  feedback, making the hotkey appear randomly broken.

## Decision

Four changes establish the new tap-lifecycle and threading contract:

**(a) Dedicated Thread + CFRunLoop for the CGEventTap source.**
`HotKeyService` creates a private `Thread` that spins its own `CFRunLoop`. The tap's
`CFRunLoopSource` is added to that loop, not to `CFRunLoopGetMain()`. This eliminates
contention with `AVCaptureSession` and any other main-thread work.

**(b) `AVCaptureSession.startRunning()` dispatched to `sessionQueue`.**
The blocking `startRunning()` call is moved to a dedicated `DispatchQueue` (serial,
background QoS). The `isRecording` flag is flipped asynchronously on the same queue after
the call returns, keeping the main actor free.

**(c) `MonitoringState` observable enum on `HotKeyService`.**
`HotKeyService` publishes `@Published var monitoringState: MonitoringState`. The type
carries three cases:
- `.stopped` — monitoring has not been started or was cleanly stopped.
- `.active` — tap is running and receiving events.
- `.failed(TapFailureReason)` — tap creation failed or was invalidated, with a typed reason
  (`.accessibilityNotTrusted` or `.tapCreationFailed`).

`MainViewModel` subscribes to `$monitoringState` and surfaces `.failed` as a visible error
status in the menu bar. This replaces the previous silent-failure behaviour.

Retry policy is changed from a hard 3-attempt limit to unbounded capped exponential backoff:
initial delay 1 s, doubling each attempt, capped at 30 s. The service retries indefinitely
until the tap is successfully created or monitoring is explicitly stopped.

**(d) `previousFlags` reset and post-timeout Fn resync.**
`stopMonitoring()` resets `previousFlags` to `.init()` so a subsequent `startMonitoring()`
begins with a clean key-flag baseline. After the backoff delay before each retry attempt,
`CGEventSource.flagsState(.combinedSessionState)` is sampled to detect whether Fn is
currently held; if it is not, any pending `.pending` or `.recording` state is cancelled
before the tap is re-created.

**(e) `PermissionManager` polls `IsSecureEventInputEnabled()` every 2 seconds.**
When secure input is detected, `PermissionManager` publishes `isSecureInputActive = true`.
`MainViewModel` observes this flag and shows an orange "安全输入已启用，热键暂不可用" status
in the menu bar. The hotkey is not disabled — the OS already suppresses it — but the user
gets an actionable explanation.

## Alternatives considered

- Share the main run loop but lower the `AVCaptureSession` queue priority — rejected; the
  root contention is blocking vs. non-blocking run-loop usage, not priority inversion.
- Replace CFRunLoop with a `DispatchSourceMachPort` based tap — rejected; CGEventTap requires
  a `CFRunLoopSource` insertion point; a Dispatch shim adds complexity without removing the
  run-loop dependency.
- Finite retry with user-facing "restart" button — rejected for #5; the accessibility
  permission dialog can be dismissed and re-granted without an app restart, so unbounded retry
  with a sensible cap is more recoverable.
- Disable hotkey when secure input is active at the OS level — not possible; `CGEventTap`
  suppression is enforced by the kernel. Detection and display is the only available action.

## Consequences

### Threading invariants (must be preserved in future changes)

| Thread / Queue | Owns |
|---|---|
| Tap thread (`CFRunLoop`) | CGEventTap callback, `CFRunLoopSource` |
| `sessionQueue` (serial background) | `AVCaptureSession.startRunning()` / `stopRunning()`, `isRecording` flip |
| `@MainActor` | `monitoringState` publish, all UI updates, `MainViewModel` state |

- Any code that touches `monitoringState` or `MainViewModel` state must run on `@MainActor`.
- Any code that calls `AVCaptureSession` session lifecycle methods must run on `sessionQueue`.
- The tap callback must never dispatch synchronously to `@MainActor` to avoid deadlock.

### Other consequences

- `monitoringState` is now the authoritative signal for tap health; callers must not poll the
  tap object directly.
- Unbounded retry means a missing Accessibility permission will produce log noise; the retry
  interval cap (30 s) bounds this to a low rate.
- `TapFailureReason` is a public type; adding new cases is a source-compatible change but
  callers with exhaustive switches must be updated.
- `IsSecureEventInputEnabled()` polling adds a 2-second background timer; it must be stopped
  in `PermissionManager.deinit` (or on app termination) to avoid a leak.
